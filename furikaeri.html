🎯 今日のゴール

「ドラクエ風のドット絵RPGを作る。主人公はAI。バトルは数学の問題を解くとレベルアップ」—— このアイデアを、スマホでも遊べるWebゲームとして公開するところまでやりました。

🗺 やったことの流れ

  • STEP 1
    ゲームを作った
    HTML / CSS / JavaScript の3ファイルで、フィールド探索＋数学バトルのRPGを実装。 ドット絵はすべてCanvasに手描き（画像ファイルなし）。
  • STEP 2
    スマホ操作に対応した
    画面タッチで動かす十字キーを追加。タッチ端末のときだけ表示されるようにした。
  • STEP 3
    インターネットに公開した
    GitHub Pages を使って、世界中からアクセスできるURLで公開。 プッシュするたびに自動で更新される仕組み（GitHub Actions）も用意。
  • STEP 4
    スマホで答えが打てない問題を解決
    iPhoneだとキーボードが自動で開かなかったため、ゲーム内に数字パッドを追加。
  • STEP 5
    レイアウトのズレを修正
    入力欄が大きすぎてボタンが見切れていたのを、1行に固定して整理。

💡 今日わかったこと

① Webページは3つの役割でできている

HTML＝中身・骨組み、CSS＝見た目・デザイン、JavaScript＝動き・ロジック。 この3つを分けて書くと整理しやすい。

② ドット絵はコードでも描ける

画像を用意しなくても、Canvasに四角を並べることでドット絵キャラを描ける。 色の対応表（パレット）と文字の地図でスプライトを表現した。

③ スマホとPCで操作が違う

PCはキーボード、スマホはタッチ。同じページでも両方に対応するため、 タッチ端末だけ十字キーや数字パッドを出す工夫をした。

④ 公開には「ホスティング」が必要

自分のPCの中にあるだけでは他の人は見られない。 GitHub Pagesに置くことで、URLでどこからでもアクセスできるようになる。

⑤ 自動デプロイは便利

コードを更新（プッシュ）すると、GitHub Actionsが自動でサイトを作り直してくれる。 手作業が減ってミスも減る。

⑥ つまずきは「原因→対策」で直す

「打てない」→iOSの仕様が原因→数字パッドを足す。 問題は原因を特定すると解決策が見える。

📖 きょうの用語メモ

デプロイ (deploy)
作ったプログラムを、実際に動く場所（サーバーなど）に置いて使えるようにすること。
GitHub Pages
GitHubのリポジトリの中身を、そのままWebサイトとして無料で公開できる仕組み。
レスポンシブ対応
画面の大きさ（スマホ／PC）に合わせて、見た目や操作を自動で変えること。
キャッシュ
ブラウザが前に見たページを一時保存する仕組み。更新が見えないときは「再読み込み」で解決。
